## 1. Goal

Port Hermes's file-backed `MemoryStore` as a **standalone, bounded, file-persisted curated memory
store** carrying two targets — **Org** (≈ `MEMORY.md`: hiring standards / rubrics / policy) and
**Recruiter** (≈ `USER.md`: a recruiter's preferences / a manager's "bar"). It maintains two
parallel states: a **frozen snapshot** (enters the system prompt, stable for the whole session) and
a **live entry list** (mutated by add/replace/remove and persisted to disk). This is the curated,
low-volume, strongly-consistent layer of the Memory Subsystem (PRD §9.3).

## 2. Scope (this point only)

In: the `MemoryStore` class with all its mechanisms, ported faithfully — two-state model,
`ENTRY_DELIMITER`, fixed char budgets, `load_from_disk`, `save_to_disk` (atomic temp→fsync→`os.replace`
under a `.lock`), `add`/`replace`/`remove` (unique-substring match), `apply_batch` (all-or-nothing),
`_detect_external_drift` (→`.bak`, reject), a pass-through write gate, `format_for_system_prompt`; the
two targets `org`/`recruiter` with configurable budgets; an injected threat-scan seam; tests; a
security-review record; a small inspect demo.

Out (later points): wiring into the agent loop (§1.3 `MemoryProvider`/`MemoryManager`); the
governance header + consent gate (§1.5); the real `threat_patterns` library (§1.6); the embedded
vector store + entity providers (§1.4).

## 3. Grounding & build strategy (TEXTBOOK_SPEC Tenet 1)

This is the **first real code port**. We port `MemoryStore` from `tools/memory_tool.py` and keep the
**core algorithms byte-for-byte faithful**: dedup (`dict.fromkeys`, order-preserving), fixed-length
budget, atomic write, `.lock` (fcntl / msvcrt), drift detection, `apply_batch` all-or-nothing, the
lean success response. `THIRD_PARTY_NOTICES.md` records this as a **Port** (was "design-derived" at
§1.1) with the MIT copyright retained.

**What changes (and only this):**
- Targets renamed `memory → org`, `user → recruiter`; files `ORG.md` / `RECRUITER.md`.
- The injection scan is an **injected dependency** `scan_entry: Callable[[str], str | None]`
  (default pass-through). The real `threat_patterns`-based scanner is plugged in at §1.6.
- Char budgets recalibrated (Org raised, e.g. 6000; Recruiter e.g. 2000) — **still bounded** (the
  fixed-length / high-SNR principle is preserved; unbounded growth would break the snapshot's
  prefix-cache benefit).
- Governance header (provenance / consent / retention) is **not** added here (entries stay opaque
  text; a header can be prefixed later without breaking `ENTRY_DELIMITER` splitting) — §1.5.

**Interface:**
```

# target ∈ {"org", "recruiter"}
```

## 5. The threat-scan seam

`scan_entry(text) -> str | None`: returns a short threat description (the entry is replaced by
`[BLOCKED: …]` in the **snapshot**, while the **live** state keeps the original for human review) or
`None` (clean). Default is a pass-through (`lambda _t: None`). §1.2 proves the `[BLOCKED:]` mechanism
with a stand-in scanner in tests; §1.6 supplies the real one. This mirrors the §1.1 pattern of
injecting a dependency and providing the real impl later.

## 6. Two-state model & data flow

- **Live**: `org_entries` / `recruiter_entries` lists, mutated by add/replace/remove/apply_batch and
  persisted via `save_to_disk` (atomic). Tool responses reflect live state.
- **Frozen**: `_system_prompt_snapshot = {"org": …, "recruiter": …}` generated once at
  `load_from_disk()` (after dedup + scan), and **never changes for the session**.
  `format_for_system_prompt(target)` returns the snapshot block (Key Invariant #1).
- **Load**: read file → split on `ENTRY_DELIMITER` → drop blanks → dedup → `scan_entry` each entry
  (hit → `[BLOCKED:]` in snapshot, original in live) → freeze snapshot.
- **Drift**: before a write, `_detect_external_drift` checks the on-disk file can round-trip and has
  no over-budget giant entry; on suspicion → write `path.bak.<ts>` and **reject the write**.

## 7. Testing & acceptance (= Plan §1.2 matrix + Exit Criteria)

Unit tests in `tests/test_memory_store.py` (tmp dirs; deterministic):
- atomic round-trip (write then read yields identical entries; no partial file left);
- lock path executes on **Windows (msvcrt)** (our platform) and POSIX — sequential adds both land;
- drift → `.bak.<ts>` created + write rejected + original content preserved;
- `apply_batch` all-or-nothing (final-budget validation; transient mid-step over-budget allowed);
- replace ambiguous-match (≥2 distinct) → error, no deletion;
- fixed-length overflow → reject + echo `current_entries`;
- injected entry → `[BLOCKED:]` in snapshot, original in live, 0 into the system prompt (stand-in scanner);
- add idempotency within `apply_batch`;
- lean success response (single add does not echo all entries).

Exit: all green on Windows + POSIX lock paths; injection-on-load → `[BLOCKED:]` in snapshot only;
drift drill rejects + `.bak`, zero loss.

## 9. Risks

- **Faithful-port drift**: re-implementing instead of porting risks subtle behaviour change → port
  method-by-method against the real source; cite symbols; the test matrix mirrors Hermes's cases.
- **Windows lock path**: must be exercised (high Windows customer share) — test the `msvcrt` branch,
  not only `fcntl`.
- **Budget recalibration** could erode the high-SNR principle → keep bounded; document the rationale.
