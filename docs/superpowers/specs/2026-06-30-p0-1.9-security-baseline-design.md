# Phase 0 ¬ß1.9 ‚Äî Security baseline (local-first): design spec

**Date:** 2026-06-30
**Workstream:** Production Plan ¬ß1.9 (Security baseline, local-first)
**Branch:** `phase0/1.9-security-baseline` (off `phase0/1.8-canonical-data-audit`)
**Cycle shape:** one cohesive cycle delivering the **two Phase-0 foundations** kept by the
2026-06-30 scope decision (Plan ¬ß1.9): the **RBAC/ABAC engine** and **at-rest encryption + OS-keystore
key handling**. SSO, secret store + rotation, field-level encryption, and app-update signing are
**deferred** (committed in `cf06b90`).

---

## 1. Goal

Deliver, behind clean seams and **without breaking the existing 214-test suite**:

1. A **RBAC/ABAC policy engine** (`security/rbac.py`) that resolves a ¬ß1.8 `User`/`Org` into a scoped
   `Principal` and authorises actions ‚Äî the **single source** the ¬ß1.5 memory-recall filter reuses
   (no permission-model drift).
2. **At-rest encryption** of the local SQLite databases (via SQLCipher) and the file-backed memory
   store (via AES-256-GCM), with the **master key held in the OS keystore** (Windows DPAPI / macOS
   Keychain), never in plaintext on disk ‚Äî so a raw-disk read yields only ciphertext.
3. The **threat-modelling v1** initial-review record over those two foundations.

Both foundations are **opt-in** (encryption gated by config, default off; RBAC default `FULL_ACCESS`)
so existing demos/tests are unaffected; the acceptance tests enable them explicitly.

---

## 2. Scope (from the committed Plan ¬ß1.9 decision)

**In scope (Phase 0):**
- `security/rbac.py` ‚Äî RBAC/ABAC engine (the ¬ß1.5 "same source").
- `security/keystore.py` ‚Äî OS-keystore master-key handling (DPAPI / Keychain / dev fallback).
- `security/cipher.py` ‚Äî AES-256-GCM data cipher from the master key.
- `security/db_encryption.py` ‚Äî `open_encrypted_db()` (SQLCipher) + the encryption-disabled passthrough.
- Wiring: `CoreConfig.encryption_enabled` (default off) + key source; route store DB connects through
  the helper; inject an optional cipher codec at the ¬ß1.2 `MemoryStore` byte boundary (port logic
  byte-unchanged when off); `.gitignore` hardening (`*.db`/`*.sqlite`).
- `docs/security/p0-1.9-threat-model-v1.md` ‚Äî STRIDE-lite over the two foundations.

**Out of scope (deferred, per the committed scope):** field-level (per-column) encryption, secret
store + rotation, SSO (OIDC/SAML), update-package signing/integrity. The interim secret path stays the
existing gitignored `.env` (¬ß1.11 will add the dedicated store).

**Invariants preserved:** no `core/agent_loop.py` change; the ¬ß1.2 ported `MemoryStore` logic is
byte-unchanged when encryption is off; the system-prompt frozen-snapshot and the ¬ß1.0 Key Invariants
are untouched.

## 3. Dependency decision (verified 2026-06-30)

Stdlib has **no symmetric cipher**, so a crypto dependency is unavoidable. Verified on this box
(Python 3.12.9, Windows 11):

| Package | Result | Use |
|---|---|---|
| `cryptography` | ‚úÖ wheels for every version (incl. win) | AES-256-GCM (`security/cipher.py`) + file-store codec |
| `sqlcipher3>=0.6.2` | ‚úÖ **prebuilt `cp312-win_amd64` wheel** downloaded under `--only-binary` (no compiler) | transparent whole-DB encryption |
| `sqlcipher3-binary` | ‚ùå does not exist as a distribution | ‚Äî (do **not** depend on it) |
| `pysqlcipher3` | ‚ùå no binary wheel (source only) | ‚Äî (do not use) |

**Decision:** depend on `sqlcipher3>=0.6.2` and `cryptography`. This satisfies the "Windows-first,
one-click install, no compiler for SMBs" constraint. **Residual:** macOS prebuilt-wheel availability is
unverified from this box ‚Äî record as a residual to confirm on a Mac (the threat-model doc + devlog note
it; it does not block the Windows-first target). Licenses recorded: `cryptography` (Apache-2.0 OR BSD),
SQLCipher (BSD-style) ‚Äî these are **dependencies, not Hermes ports**, so they go in a dependency-license
note, not `THIRD_PARTY_NOTICES.md` (which is for copied source).

## 4. Architecture & components

New package surface under `agent/src/jobpin_agent/security/` (joins the existing `threat_patterns`,
`scrubber`, `external_ingest`):

### 4.1 `security/keystore.py` ‚Äî master-key handling (zero new deps)

```
class KeyStore(ABC):
    def get_or_create_master_key(self) -> bytes        # 32 bytes; created once, then protected at rest

class WindowsDpapiKeyStore(KeyStore):                  # ctypes -> CryptProtectData/CryptUnprotectData
    # wrapped key stored in a local file; on-disk bytes are DPAPI ciphertext (user/machine-bound)
class MacKeychainKeyStore(KeyStore):                   # subprocess -> `security add/find-generic-password`
class DevKeyStore(KeyStore):                           # env/file; CLEARLY insecure; CI/Linux/tests only

def default_keystore() -> KeyStore                     # picks by platform; DevKeyStore fallback
```

- Master key = `secrets.token_bytes(32)`, generated once, then **protected by the backend** (DPAPI
  wraps it / Keychain stores it). The plaintext master key never touches disk.
- `DevKeyStore` is explicitly insecure (reads a fixed env var / file) so CI and Linux can exercise the
  encryption path; the threat-model doc states it must not be used in production.

### 4.2 `security/cipher.py` ‚Äî data cipher (`cryptography`)

```
class Cipher:
    def __init__(self, key: bytes)                     # AES-256 key (derived from master key)
    def encrypt(self, plaintext: bytes) -> bytes       # returns nonce(12) || ciphertext || tag
    def decrypt(self, blob: bytes) -> bytes            # raises on tampering/wrong key (AEAD)
```

- AES-256-GCM via `cryptography`'s `AESGCM`. Random 12-byte nonce per call, prepended.
- Decrypt **raises** (`cryptography.exceptions.InvalidTag`) on tamper/wrong-key ‚Äî never returns garbage.

### 4.3 `security/db_encryption.py` ‚Äî encrypted SQLite open

```
def open_encrypted_db(path: str, key: bytes | None) -> Connection
    # key is not None  -> sqlcipher3.dbapi2.connect(path); conn.execute("PRAGMA key = ...")
    # key is None      -> sqlite3.connect(path)            (encryption disabled / tests / :memory:)
```

- Returns a DBAPI-compatible connection either way, so the stores' existing SQL is unchanged.
- The PRAGMA key is the hex of a DB-subkey derived from the master key (HKDF or a labelled hash) so the

### 4.4 `security/rbac.py` ‚Äî RBAC/ABAC engine (zero new deps)

```
@dataclass(frozen=True)
class Role:
    name: str
    permissions: frozenset[str]          # action verbs: read/write/delete/export/configure
    scope_templates: tuple[str, ...]     # e.g. "{tenant}:{org}:candidate"; "" => everything (system)
    sensitivity_ceiling: str             # "normal" | "sensitive"

ROLE_POLICIES: dict[str, Role]           # starter table: admin / recruiter / interviewer / system
                                         # data-driven so new roles are config, not code

@dataclass(frozen=True)
class Decision:
    allowed: bool
    reason: str                          # "ok" | "rejected:rbac" (matches ¬ß1.8 audit vocabulary)

def principal_for(user: User, org: Org) -> Principal
    # fills the role's scope_templates with user.tenant_id/org_id -> Principal(allowed_scopes=...)
    # org/tenant isolation is STRUCTURAL: a role in org A can only be handed org-A scopes
    # unknown role -> a no-scope Principal (deny-all), never an exception that crashes the caller

def authorize(principal: Principal, action: str, resource: ResourceRef) -> Decision
    # RBAC: role has `action` permission for resource.type
    # ABAC: resource.org_id == principal org  AND  resource.sensitivity <= role.sensitivity_ceiling
    #       AND  scope_predicate(principal)(resource.memory_key)
    # FAILS CLOSED: unknown role / missing policy / out-of-scope -> Decision(False, "rejected:rbac")

@dataclass(frozen=True)
class ResourceRef:
    type: str            # candidate/job/...
    org_id: str
    memory_key: str
    sensitivity: str = "normal"
```

- **Same source:** `Principal` and `scope_predicate` remain importable from `governance/rbac.py`
  (only two test files import them; no production module hard-imports `Principal`). `security/rbac.py`
  imports `Principal`/`scope_predicate`/`FULL_ACCESS` from `governance/rbac.py` and is the **sole place
  that derives scopes from a user** and authorises actions. The recall path is unchanged ‚Äî it still
  takes a `Principal` and applies `scope_predicate`; only *who builds the Principal* changes (the
  engine, when a real composition root exists). Default stays `FULL_ACCESS`.
  - *(Layering note: `governance/rbac.py` has no imports that could cycle with `security`; the import
    direction security‚Üígovernance is safe. A future move of the model into `security/` with a
    governance re-export is possible but unnecessary now and adds churn ‚Äî deferred.)*

### 4.5 `core/config.py` (edit)

Add `encryption_enabled: bool = False` and a key-source selector (env `JOBPIN_ENCRYPTION=1`, keystore
chosen by platform). Default off keeps the suite green. This is the **composition seam** ‚Äî the stores
read the cipher/connection from config-driven wiring; there is still no full app entry point (consistent
with the ¬ß1.1 note), so the wiring is exercised by tests, not a `main`.

## 5. Data flow

- **Auth:** request ‚Üí `principal_for(user, org)` ‚Üí `Principal` ‚Üí
  - memory recall: `scope_predicate(principal)` filters `memory_key`s **before** top-k (no-leak);
  - actions: `authorize(principal, action, resource)` ‚Üí `Decision`; a `rejected:rbac` denial is recorded
    via the ¬ß1.8 `AuditStore` (the action/result vocabulary already exists there).
- **Crypto:** store write ‚Üí `cipher.encrypt(bytes)` ‚Üí ciphertext on disk; read ‚Üí `cipher.decrypt`.
  SQLite DBs ‚Üí SQLCipher `PRAGMA key`. Master key ‚Üí keystore (DPAPI-wrapped / Keychain), never plaintext.

---

## 6. Error handling ‚Äî fail closed / fail safe

- **RBAC** denies on unknown role, missing policy, out-of-scope, or sensitivity breach
  (`Decision(False, "rejected:rbac")`); `principal_for` on an unknown role returns a deny-all Principal,
  not an exception.
- **Crypto** raises on a missing key or decrypt failure (AEAD `InvalidTag`); it **never** silently falls
  back to writing/reading plaintext. A drift/round-trip failure in the ¬ß1.2 store keeps its existing
  reject-and-backup behaviour.
- **Encryption disabled** (default) is a clean passthrough ‚Äî `open_encrypted_db(path, None)` is plain
  `sqlite3`; the file-store codec is identity. No behavioural change when off.

## 7. Testing & acceptance

Maps to the Plan ¬ß1.9 exit criteria.

**Encryption (exit criterion 1 ‚Äî "raw-disk read cannot yield plaintext"):**
- `cipher` round-trip; tamper/wrong-key ‚Üí raises.
- `open_encrypted_db` with a key: write a row, **read the raw file bytes off disk**, assert the known
  plaintext (e.g. a candidate name) is **absent** (ciphertext); reopen with the right key ‚Üí row returns;
  wrong key ‚Üí open/read fails.
- keystore: `get_or_create_master_key` is stable across calls; the on-disk wrapped-key bytes do **not**
  contain the raw key (DPAPI/Dev). (`DevKeyStore` used in CI; DPAPI path tested on Windows.)
- file-store codec round-trip; with encryption off, the ¬ß1.2 store output is byte-identical to today.

**RBAC (exit criterion 2 ‚Äî "RBAC available; passes an unauthorised-access test"):**
- `principal_for`: each role ‚Üí expected `allowed_scopes` (templated with org/tenant); a user in org A
  never receives an org-B scope.
- `authorize` matrix: cross-org read ‚Üí `rejected:rbac`; a role lacking `export` ‚Üí denied; sensitivity
  ceiling breach ‚Üí denied; in-scope permitted action ‚Üí `ok`.
- `scope_predicate` reuse: a recruiter principal's predicate excludes out-of-scope keys (no-leak),
  reusing the existing ¬ß1.5 function.

**Regression:** `python -m pytest agent` stays green with encryption off (target: prior 214 passed,
2 skipped, plus the new ¬ß1.9 tests). The 2 money-safe OpenAI skips remain skipped.

**Threat-model v1:** `docs/security/p0-1.9-threat-model-v1.md` exists and covers assets, the STRIDE-lite
threats, the at-rest + RBAC controls, and residual risks/deferrals.
