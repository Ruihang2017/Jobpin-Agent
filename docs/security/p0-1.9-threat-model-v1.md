## Assets being protected
1. **Candidate / employee PII** at rest — the canonical store (`data/store.py`), the §1.4 structured + vector stores, session history.
2. **Audit trail** (§1.5/§1.8 `AuditStore`) — shares the canonical connection, so it is encrypted at rest when the store is keyed.
3. **Org / recruiter memory** — the file-backed `MemoryStore` (ORG.md / RECRUITER.md).
4. **The master key** — the root of all the above; must never be at-rest plaintext.

## Security-review checklist

| # | Concern | Finding |
|---|---|---|
| 1 | **Crypto primitive soundness** | AES-256-GCM via `cryptography`'s `AESGCM`; fresh 12-byte `os.urandom` nonce per encrypt, prepended; 32-byte key enforced. HKDF-SHA256 derives labelled subkeys (`db` vs `file`) so one master key protects both surfaces without sharing key material. No home-rolled crypto. ✅ |
| 2 | **No weak KDF / no SQL-injection in PRAGMA** | SQLCipher is keyed with the raw-key blob form `PRAGMA key = "x'<64-hex>'"` (hex of a derived 32-byte subkey) — no passphrase KDF, and the value is hex only (no injection surface). ✅ |
| 3 | **Fail-safe: no silent plaintext** | When a key is supplied, `import sqlcipher3` failing **raises** (no fallback to plaintext). `decrypt` raises on wrong key / tamper. `CoreConfig` refuses to start if encryption is requested but unwired. ✅ |
| 4 | **Fail-closed RBAC** | Every deny path returns `rejected:rbac`; unknown role/sensitivity default to the most restrictive; the `:`-delimited scope prefix check prevents prefix-confusion (`apac` vs `apac2`). ✅ |
| 5 | **"Same source" — no permission-model drift** | One `Principal` + one `scope_predicate` (in `governance/rbac.py`); `security/rbac.py` is the sole *deriver* of scopes and layers role/action/sensitivity on top of the shared key predicate. ✅ |
| 6 | **§1.2 port off-path unchanged** | Every file-store encryption branch is guarded `if cipher is None: <original code>`; the off-path (incl. a non-UTF-8 file re-raising) is byte-identical — proven by the unchanged §1.2 suite. ✅ |
| 7 | **No plaintext leak on the drift `.bak`** | Under encryption, `_backup_drift` snapshots the on-disk **ciphertext** bytes, never the decrypted text — `test_drift_backup_no_plaintext_leak_under_encryption`. ✅ |
| 8 | **DPAPI ctypes correctness** | Input buffer kept alive across the call; output freed with `LocalFree`; `WinError` on failure. `test_dpapi_roundtrip` green on this Windows box. ✅ |
| 9 | **No dynamic code / unsafe deserialisation** | No `eval`/`exec`/`pickle`/`yaml`. `subprocess` only for the macOS `security` CLI with a fixed argv (no shell). ✅ |
| 10 | **Dependency hygiene** | Only `cryptography` + `sqlcipher3` added; both ship prebuilt wheels for the Windows-first target (verified). Licences recorded above. `sqlcipher3` is imported lazily inside the keyed branch, so the encryption-off / `:memory:` path never loads it. ✅ |
| 11 | **Honest scope (no overclaim)** | Stated plainly: encryption is opt-in and **not yet reachable from a running path** (no composition root — the flag fails loud); RBAC is not yet wired into live recall; the macOS Keychain path is untested (no Mac in CI); the deferred items are absent, not stubbed-as-done. ✅ |

## Residual risks / deferrals (tracked)
- **`DevKeyStore` is insecure by design** (XOR with a public pad) and `default_keystore` falls back to it on any non-Windows/non-macOS platform (e.g. Linux/CI) **silently**. It exists only so the encryption path can be exercised in CI. **Must not be used in production.** A production Linux target needs a real backend (libsecret/SecretService) and the silent fallback should become an explicit opt-in or a startup warning — deferred with the Linux production target.
- **Encryption is dormant.** `CoreConfig.encryption_enabled`/`master_key_path` are a seam; no composition root turns the keystore master key into store `cipher_key`s yet (the standing §1.1 gap). The flag fails loud until wired. End-to-end keystore→cipher→store is proven by `test_keystore_to_store_end_to_end`, but no *running* path enables it.
- **macOS Keychain backend untested** — no Mac in CI; the macOS `sqlcipher3` prebuilt-wheel availability is also unverified from the Windows dev box. Both are residuals to confirm on a Mac.
- **Audit has no cryptographic tamper-evidence** (append-only at the API level only); crypto chaining is a Phase-2 item.
- **Deferred §1.9 items** (per the committed scope): field-level encryption, secret store + rotation, SSO, update-package signing — each with a trigger in the Plan.
- **Compliance framing**: at-rest encryption is a **serious-harm-mitigating control** under the NDB scheme, **not** a statutory safe harbour (the §1.9 contract + PRD §11.5 wording reconciled this cycle).

## Conclusion
**Cleared for §1.9 scope.** Both at-rest surfaces (SQLite via SQLCipher, files via AES-256-GCM) demonstrably
yield ciphertext on a raw-disk read; the master key is OS-keystore-protected and never at-rest plaintext;
the RBAC/ABAC engine fails closed and is the single §1.5 source; the §1.2 port off-path is unchanged; no
plaintext leaks to the drift `.bak`; and the dormant encryption flag fails loud rather than lulling an
operator into a false sense of security. The net-new crypto uses audited primitives with no home-rolled
algorithms. The residuals above (DevKeyStore-not-for-prod + silent Linux fallback, the dormant flag until a
composition root lands, the untested macOS path, audit tamper-chaining) are tracked, not hidden.
