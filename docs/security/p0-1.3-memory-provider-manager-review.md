## Why this review exists
MIT grants the code "as is" with no warranty. A regulated, local-first HR product cannot blindly trust
third-party code, so each ported file gets its own review (TEXTBOOK_SPEC; PRD §2.7). This covers the §1.3
port (orchestration + provider contract + fence), not the deferred write/scan/streaming pieces.

## Checklist

| # | Concern | Finding |
|---|---|---|
| 1 | **Background-thread safety** | The sync/prefetch executor is a single `ThreadPoolExecutor(max_workers=1)` created lazily under `_sync_executor_lock`. One worker serialises writes (turn N before N+1); `flush_pending` submits a sentinel and waits (deterministic barrier). No shared mutable state is written from the worker except each provider's own internals (providers are responsible for their own thread-safety). ✅ |
| 2 | **Teardown cannot hang** | `_drain_sync_executor` shuts the executor down with `cancel_futures=True`, then waits at most `_SYNC_DRAIN_TIMEOUT_S` (5.0s) on a **daemon** watcher thread. A wedged provider cannot block process/session teardown. **Note (divergence from Hermes's comment):** modern `ThreadPoolExecutor` workers are non-daemon with an `atexit` join, so a truly-wedged in-flight task can still delay *interpreter* exit; the bounded drain only guarantees `shutdown_all` itself returns promptly. Tracked as a known limitation; acceptable for local-first (the worker does memory I/O, not unbounded network in §1.3). ✅ (bounded) |
| 3 | **Failure isolation** | Every provider hook call (`system_prompt_block`/`prefetch`/`sync_turn`/`queue_prefetch`/`on_*`/`get_tool_schemas`/`handle_tool_call`/`initialize`/`shutdown`) is wrapped in try/except + `logger`. One provider raising never blocks the others or the turn. Covered by `test_failure_isolation_one_provider_raises`. ✅ |
| 4 | **Prompt-injection via recalled memory** | Recall is wrapped in a `<memory-context>` fence + a system note ("NOT new user input … reference data"); `sanitize_context` strips any fence tags / blocks / notes a provider smuggles, so a provider (or recalled resume text) cannot forge "authoritative" framing or break out of the fence. The §1.1 loop owns the outer tags; the seam returns the inner block (no double-wrap). Covered by `test_memory_fence.py` + `test_prefetch_wrapped_then_fence_stripping`. The **real** `threat_patterns` scan of external text is §1.6 — until then, the fence is structural containment, not content scanning. ⚠️ (by design) |
| 5 | **Fence regex / ReDoS** | The three ported regexes use bounded/lazy forms (`[\s\S]*?` lazy; a character-class note matcher `[^\]]*`; a simple tag matcher). No nested quantifiers over overlapping alternations → no catastrophic-backtracking vector on adversarial input. Inputs are bounded by the curated budget / provider output. ✅ |
| 6 | **Tool routing cannot be hijacked** | A provider tool whose name is in `_CORE_TOOL_NAMES` (`clarify`/`delegate_task`) is dropped at registration and never advertised or routed (built-ins always win); a duplicate tool name keeps the first registrant. Nameless/malformed schemas are skipped by `normalize_tool_schema` (prevents the HTTP-400 "missing name" poisoning). Unknown tools route to a `tool_error` JSON, never an exception to the model. Covered by `test_core_tool_not_shadowed` / `test_tool_routes_to_owning_provider`. ✅ |
| 7 | **Single-external-provider rule** | `add_provider` accepts `builtin` always and rejects a second non-builtin with a warning — bounding tool-schema surface and backend conflict (`CompositeMemoryProvider` merging is Phase 2). Covered by `test_second_external_provider_rejected`. ✅ |
| 8 | **No dynamic code / unsafe deserialisation** | No `eval`/`exec`/`pickle`/`yaml`/`subprocess`. `json.loads` is used only in `_memory_tool_result_succeeded` (fail-closed: non-JSON / non-dict / missing `success` / `staged` → treated as "did not commit"). `inspect.signature` is read-only introspection. ✅ |
| 9 | **Write mirroring is inert + fail-closed in §1.3** | `notify_memory_tool_write` / `on_memory_write` are ported but only fire for an **external** provider (skips `builtin`) and only on a committed (non-staged, `success:true`) write. With no external provider and no model-facing write tool yet (→§1.5), this path is dormant. No ungoverned write path is exposed in §1.3. ✅ |
| 10 | **Transitive dependencies** | Standard library only (`concurrent.futures`, `threading`, `inspect`, `json`, `re`, `logging`, `dataclasses`, `typing`). No third-party imports. ✅ |
| 11 | **Seam adapter does not leak / mutate §1.1** | `MemoryManagerHooks` only reads the message list (last user/assistant text + a dict render) and delegates; it never mutates `agent_loop.py` state or the frozen system prompt. `build_memory_backend` is an assembly helper, not an app entry point. ✅ |

## Conclusion
The §1.3 port is faithful and self-contained. **Cleared for §1.3 scope.** Tracked dependencies that
complete the security posture: the governed write gate + model-facing write tool (§1.5, so writes are
rejected without provenance/consent labels — Key Invariant #4), and the real `threat_patterns` content
scan + `StreamingContextScrubber` (§1.6). Item #2's interpreter-exit caveat is a known limitation, not a
§1.3 blocker.
