## Why this review exists
MIT grants the code "as is" with no warranty. A regulated, local-first HR product
cannot blindly trust third-party code, so each ported file undergoes its own
review (TEXTBOOK_SPEC; PRD §2.7). This covers the §1.2 port only.

## Checklist

| # | Concern | Finding |
|---|---|---|
| 1 | **Path handling / traversal** | `memory_dir` is **caller-supplied** (no env/global); files are fixed names `ORG.md`/`RECRUITER.md` joined to it (no user-controlled path segments). The `.lock` / `.bak.<ts>` / `.tmp` siblings stay inside `memory_dir`. No traversal vector. ✅ |
| 2 | **Atomic write** | `_write_file` writes a temp file in the same dir → `fsync` → `os.replace` (atomic on one filesystem). A reader sees the complete old or new file, never a truncated one. Temp file cleaned on any failure. ✅ |
| 3 | **Lock file** | `_file_lock` locks a **separate** `.lock` file (so the data file can still be atomically replaced), with `fcntl` (POSIX) / `msvcrt` (Windows) and a no-op fallback. Unlock + close in `finally`. Windows path is **tested** (`test_lock_path_executes_on_this_platform`). ✅ |
| 4 | **Drift `.bak`** | `_detect_external_drift` writes the snapshot to `path.bak.<ts>` **inside `memory_dir`** and refuses the mutation, preventing silent data loss. The `.bak` is a copy of already-on-disk content (no new secret exposure). ✅ |
| 5 | **Injection on load** | A poisoned-on-disk entry is replaced by `[BLOCKED:]` in the **snapshot** (what enters the system prompt) while the **live** state keeps the original for human removal. In §1.2 the scan is the injected `scan_entry` (**default pass-through**), so blocking is only as strong as the injected scanner. **The real `threat_patterns` library is a §1.6 dependency — tracked.** Until §1.6, do not rely on this store to block injection. ⚠️ (by design) |
| 6 | **Scan/write-gate seams** | Both default to safe-by-omission: `scan_entry` default = no block (explicit, documented); `write_gate` default = pass-through. Neither executes untrusted input. ✅ |
| 7 | **No dynamic code / unsafe deserialisation** | No `eval`/`exec`/`pickle`/`yaml`/`subprocess`. Entries are opaque text; files are plain UTF-8 split on a delimiter. ✅ |
| 8 | **Resource / DoS** | Fixed char budgets bound store size; `apply_batch` validates the final budget; drift guard rejects over-budget giant entries. Reads are whole-file (bounded by the budget for tool-written files; an external giant file triggers drift, not unbounded processing of a committed write). ✅ |
| 9 | **Transitive dependencies** | Standard library only (`os`, `tempfile`, `fcntl`/`msvcrt`, `contextlib`, `pathlib`, `time`, `logging`). No third-party imports. ✅ |
| 10 | **Error handling** | File I/O wrapped (`OSError`/`IOError` → empty/refuse, never crash the agent); write failure raises `RuntimeError` with context after cleaning the temp file. ✅ |

## Conclusion
The port is faithful and self-contained; the algorithms (dedup, fixed-length
budget, atomic write, `.lock`, drift detection, `apply_batch` all-or-nothing) are
unchanged from the reviewed Hermes source. The single open item is **expected and
tracked**: injection blocking depends on the real `threat_patterns` scanner, which
is ported and wired in at **§1.6**. Until then the default `scan_entry` does not
block, and the `[BLOCKED:]` *mechanism* is proven with a stand-in scanner in tests.
**Cleared for §1.2 scope.**
