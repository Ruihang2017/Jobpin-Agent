## Why this review exists
§1.4 is the first large-volume memory layer (it will hold candidate PII derivatives — résumé vectors +
structured fields). A regulated, local-first HR product reviews every memory component before trusting
it. This covers §1.4's stores + providers; the deferred governance/RBAC/backends are tracked.

## Checklist

| # | Concern | Finding |
|---|---|---|
| 1 | **SQL injection** | Every query in `vector/store.py` + `structured.py` is **parameterised** (`?` placeholders); no string-built SQL. Table/column names are literals. ✅ |
| 2 | **Prefix-match safety** | `delete_by_key_prefix` / `search(key_prefix=…)` use `memory_key = ? OR memory_key LIKE ? ESCAPE '\'` with LIKE wildcards (`% _ \`) escaped in the literal prefix, so a crafted key can't widen the match. Exact-or-nested (`prefix:%`) only. ✅ |
| 3 | **Retrieve-then-filter leak** | The Candidate provider applies `scope_filter` (RBAC/namespace) to the **structured store FIRST**, then runs vector NN **only over the allowed `memory_key`s** (per-key `search`). An unauthorised candidate's vectors are never scored or returned — closing the "existence leak" (Plan §1.4/§1.5). The real RBAC policy is §1.5; §1.4 ships the seam (default open) wired in the correct order. ✅ (seam) |
| 4 | **Ungoverned writes before §1.5** | Candidate `ingest` runs through an injected `write_gate` (pass-through default; the real reject-unlabelled gate is §1.5). No model-facing write tool is exposed (none of the §1.4 providers return tool schemas). Same discipline as the §1.3 write-tool deferral — no ungoverned write path opened early. ✅ (seam) |
| 5 | **Erasure cascade** | `delete(memory_key)` removes the structured row **and** the derived vectors (`delete_by_key_prefix` on both stores) — the bounded mechanism §1.5 erasure will call. Deletion is exact-or-nested-prefix scoped (no over-deletion). ✅ |
| 6 | **Vector-space integrity (drift)** | `SqliteVectorStore` pins one `embed_version` and **rejects** a record from a different space (`ValueError`); the re-embed migration writes to a **separate** store and only switches when complete + validated, so retrieval never mixes versions. ✅ |
| 7 | **No dynamic code / unsafe deserialisation** | No `eval`/`exec`/`pickle`/`yaml`/`subprocess`. Embeddings are JSON lists of floats (`json.loads`); `text` is opaque. The embedder is pure arithmetic over hashed tokens. ✅ |
| 8 | **The fake embedder is NOT a control** | `hashing_embedder` is a lexical-overlap stand-in for *plumbing/tests*; it is not a quality or security control, and recalled résumé text is **untrusted** — the content threat scan (`threat_patterns`) + the `<memory-context>` fence that contains it are §1.3 (fence, already applied to recall) / §1.6 (scan). Documented; no security claim rests on the fake. ⚠️ (by design) |
| 9 | **Resource / DoS** | Brute-force cosine is O(n) per query at the stated hundreds–thousands scale (bounded; the benchmark surfaces the ceiling; an indexed backend is the §1.12 swap). `text` is kept inline but bounded by ingest size. The per-allowed-key search loop is bounded by the authorised set. ✅ (scale-appropriate) |
| 10 | **Transitive dependencies** | **Stdlib only** (`sqlite3`, `json`, `math`, `hashlib`, `re`, `uuid`, `time`, `dataclasses`, `typing`). No third-party import added. ✅ |
| 11 | **Composite does not weaken §1.3 invariants** | The minimal Composite is the **sole external** provider (Manager `add_provider` rule unchanged; a second external still rejected); it runs inside the §1.3 single-worker / `flush_pending` / bounded-drain machinery and isolates a failing sub-provider in `prefetch`. ✅ |

## Conclusion
**Cleared for §1.4 scope.** The stores use parameterised SQL + escaped prefix matching; retrieval
filters before NN (no existence leak); writes go through a pass-through gate (no ungoverned path);
erasure cascades; the vector space cannot silently mix. Tracked dependencies that complete the posture:
the §1.5 governance write gate + RBAC `scope_filter`, the §1.6 content threat scan, the real vector
backend (§1.12), and the real semantic embedder (config). The fake embedder's non-security status is
documented (item #8).
