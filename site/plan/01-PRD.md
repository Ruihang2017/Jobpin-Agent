# Jobpin Agent — AI 智能体驱动的 HR 平台
## 产品需求文档（PRD）

> **Jobpin Agent**。基于 Hermes Agent 的内核与**记忆系统**重新构建的、面向**澳大利亚市场**（先内部试点、后商用）的 **本地优先（local-first）** AI 原生人力资源平台。
>
> 覆盖员工全生命周期五大模块：**简历匹配、人才搜索、招聘、培训、监督与 KPI/考勤**。
>
> **合规范围声明**：本系统**仅在澳大利亚开展试点**，故本文档**只考虑澳大利亚法律法规**（联邦 + 相关州/领地）。任何向澳大利亚以外扩张（含对外商用后的跨境部分）须另行评估目标法域合规，当前不在范围内。本文档对法律的描述为产品设计输入，**不构成法律意见**，最终须经澳大利亚法律顾问确认。
>
> **叙述视角**：本文档以第三人称中立视角撰写，主体统一称"本产品 / Jobpin Agent / 系统"。

## 1. 执行摘要（Executive Summary）

**愿景**：把 HR 从"流程操作者"升级为"决策指挥者"。Jobpin Agent 是一个以 **Agent + 长期记忆** 为核心的 **本地优先** HR 平台——不只是问答机器人，而是能够跨会话、跨实体**记住**组织的招聘标准、候选人历史、员工成长轨迹，并在人类监督下自动执行多步骤工作流（搜寻、筛选、安排、培训、追踪）的智能体系统。

**差异化（相对传统 HR SaaS / 通用 Copilot）**：
1. **记忆 = 可持续壁垒（诚实表述）**：记忆**引擎**本身开源（Hermes，MIT，竞品同样可取用）——属基础能力而非护城河。真正可持续的壁垒是三者叠加：① **每客户长期沉淀的组织记忆**带来的转换成本（注意：本地优先 = 单租户，**无跨客户数据飞轮**，是 per-customer 锁定而非市场级网络效应）；② **内建并经法律审定的澳洲合规内容**；③ **全生命周期工作流深度**。机制与冷启动详见 第 9.4 节。
2. **Agent 原生**：不是把 LLM 当搜索框，而是把每个 HR 子任务交给可委派、可观测、可审计的子代理（sourcing / screening / scheduling / training / KPI agent）。
3. **本地优先、隐私优先**：商用 MVP 的 agent 运行时、记忆与 HR 数据默认运行/存储在**客户本地**，候选人/员工数据默认**不离开客户机器**。这既是强隐私卖点（HR 数据高度敏感），也天然满足澳洲数据驻留、规避 APP 8 跨境披露问题。Hermes 本就是文件型本地记忆设计，与此天然契合。
4. **内嵌专业 HR 与合规（"HR-in-a-box"）**：很多中小企业/初创**养不起专职 HR、不熟专业招聘流程与合规**。本产品把"专业 HR 流程 + 澳洲合规护栏"作为**内建的组织/程序性记忆**，用通俗语言引导非专业用户**安全、合规、低门槛**地完成 HR 工作——这是最尖锐的核心痛点，也是主要营销方向（设计要求见 第 11.8 节）。

**目标客户（一个引擎、两类客户）**：① **已有 HR 团队**的企业 → 增强效率与一致性；② **没有专职 HR** 的中小企业/初创 → 系统**充当其 HR 职能**。后者痛点最尖锐、付费意愿强，是主要商用方向。

**商业路径**：先作为企业内部工具（本地/本地部署）打磨真实场景与 ROI，再产品化为**面向"无 HR"中小企业的本地优先自助产品**对外商用（澳洲市场优先）。可选的云端/托管多租户形态作为后续、面向有此需求客户的扩展（非默认形态）。

**MVP 一句话定义**：**M1+M2+M3 的本地、单租户、强 HITL 应用**（3 个子代理 Sourcing/Screening/Scheduling + 自研轻量状态机）；**M4 培训、M5 监督考勤排在 MVP 之后**（见落地计划分期）。文中"五模块全要"指**最终范围**，非 MVP 范围。

## 2. 智能体定位与架构选型背景（Agent Vision & Architecture Rationale）

> 本章回答：**要做一个怎样的智能体？为什么借鉴 Hermes 架构？为什么不直接用 Claude Code 的架构？用/不用哪些现有框架（如 LangChain）？以及主干怎么"拥有"（移植 vs 重写）？** 结论结合当前（2025–2026）智能体技术与应用发展给出，并将在落地计划 Phase 0 通过技术验证（spike）复核。

### 2.1 本产品的智能体定位

Jobpin Agent 不是"带 HR 知识的聊天机器人"，而是一个 **有状态、长程、记忆驱动、可审计的多智能体系统**，运作范式为 **感知 → 规划 → 行动 → 记忆（在人类监督下）**：

| 维度 | 传统 Chatbot / RAG 问答 | Jobpin Agent 智能体系统 |
|---|---|---|
| 状态 | 无状态、单轮问答 | **有状态、长程**（一个招聘 loop 跨天/跨人持续推进） |
| 记忆 | 仅当前上下文窗口 | **跨会话、跨实体的长期记忆**（候选人/员工/组织/招聘官） |
| 行动 | 只生成文本 | **调用工具**（查 ATS、解析简历、排面试、写记忆），多步骤编排 |
| 结构 | 单模型 | **编排者 + 专职子代理**（sourcing/screening/scheduling/training/KPI） |
| 决策 | 黑箱、一步出结果 | **人在回路 + 可解释 + 接地引用**，每步可追溯 |
| 领域 | 通用 | 在通用 Agent 内核上做 **HR 领域专精** |

一句话定位：**在一个可被完全拥有、审计与治理的通用 Agent 内核之上，叠加"组织记忆"这一护城河，构建面向 HR 全生命周期的专精多智能体。**

### 2.2 当前智能体的发展背景（2025–2026）

技术演进已从 **"链式/RAG 问答"** 走向 **"自主智能体"**，再走向 **"有状态、带记忆层、用协议互联、可观测可治理的多智能体系统"**。对本项目的关键判断：

- **记忆正成为独立的架构层**。Mem0 / Zep / Letta / Honcho 等"记忆中间件"的兴起说明：长期记忆不再是某框架的附属功能，而是值得单独设计与治理的一层。Hermes 恰好把记忆做成了可插拔、带生命周期与安全治理的子系统——与行业方向一致，且正是 HR 产品的价值所在。
- **协议正在标准化**。**MCP** 正成为"工具/上下文接入"的事实标准；**A2A** 用于智能体间通信。新系统应面向 MCP 设计集成层，避免每个集成都写私有胶水。
- **框架会带来抽象成本**。生产实践（及 Anthropic《Building Effective Agents》的指导）反复表明：**优先用简单、可组合、透明的模式**，只在确有收益处引入重框架。在**受监管、需逐步可审计**的 HR 领域，"把 agent 循环藏进黑箱抽象"的框架是负担而非助力。
- **生产级智能体 = 控制 + 可观测 + 评测 + 人在回路**。Checkpointing、追踪、eval 门禁、HITL 中断点，是把 demo 变成可上线系统的关键。

**本项目的下注**：在 HR 这种高风险、强监管、强信任要求、且**本地优先**的场景，**记忆 + 治理 + 控制力**才是可持续价值与壁垒——架构选择应最大化这三者，而非最快搭出一个多智能体 demo。

### 2.3 为什么借鉴 Hermes 的架构

> **Hermes 是什么**：一个开源的智能体运行时/CLI（由 Nous Research 发布，**MIT 许可**），显著特点是**可插拔、带生命周期与安全治理的文件型记忆子系统**。本产品借鉴/移植其内核与记忆代码，而非作为终端用户工具使用。

- **它把最难的部分做好了，且是可拥有的透明代码**：记忆子系统（可插拔 Provider、prefetch/sync 生命周期、注入防御、上下文压缩）、子代理委派、provider 无关的模型层、清晰的模块边界。这些"地基"自研成本极高，尤其是**记忆治理**。
- **记忆是护城河**，而 Hermes 的记忆架构是其最成熟的部分（详见 第 9 节）。直接移植让本产品在差异化资产上**领先起跑**。
- **文件型本地记忆**：Hermes 的内置记忆是本地文件（原子写、文件锁、漂移检测），**与本产品"本地优先"目标天然契合**——无需为 MVP 引入云数据库。
- **Provider 无关（多模型）**：天然支持**本地模型**或可切换的模型后端，不被单一供应商锁定。
- **掌控整条主干**：在受监管领域可做到**完全可审计**、无框架锁定、可按 HR 需求自由改造。

### 2.4 为什么不直接采用 Claude Code（Claude Agent SDK）的架构

> 澄清：**"不用 Claude Code 架构" ≠ "不用 Claude 模型"。** 本产品仍可用 Claude 作主力模型，并借鉴 Claude Code 的若干优秀模式（文件型记忆"冻结快照"、MCP、子代理、hooks）。

Claude Code 的架构现以 **Claude Agent SDK** 形式提供，是生产级、成熟度很高的框架（业界生产就绪度常列第二，且正是 Claude Code 的底层）。但**不宜作为本产品主干**：

1. **Anthropic / Claude 强耦合**：以 Claude 为中心，与"模型可切换 + **本地模型** + 规避单供应商锁定"冲突。
2. **血统是"编码/终端 harness"**：为软件工程任务（文件编辑、bash、代码搜索、CLI 交互）优化，而本产品要的是**多角色门户 + 本地/本地部署后端 + 大规模按实体记忆 + HR 领域工作流**，产品形态不同。
3. **把产品建在别人的运行时之上**会把发布节奏、运维与可审计性绑定到该 harness 的约束与升级周期上；受监管、长生命周期产品更需要"拥有主干"。

**结论**：借鉴其模式与模型，但**不以其为主干**；主干用可拥有的 Hermes 派生内核。

### 2.5 现有智能体框架取舍（用什么 / 不用什么 / 为什么）

总原则：**拥有主干（Hermes 派生内核 + 记忆）→ 协议标准化（MCP）→ 边缘按需引入专用库（RAG / 编排 / 可观测 / 评测）→ 拒绝"遮蔽 agent 循环"的单体框架。** 所有取舍在 Phase 0 用 spike 验证。

> 下表横向对比为**截至 2025–26 的公开评测/社区现状**（会随版本演进），具体取舍以 Phase 0 spike 实测为准，不作为既定结论。

| 框架/方案 | 强项 | 取舍与理由 |
|---|---|---|
| **LangGraph** | 生产就绪度最高：显式状态图、checkpointing/时间旅行、HITL 中断点、可观测、模型无关 | **边缘选用（候选）**：仅用于确有"持久化、可审计、带人工卡点的状态机"处（如 M3 招聘流程）。**不作主干** |
| **LangChain（经典）** | 生态/集成最广、上手快 | **不作主干**：以"抽象泄漏/频繁变更/过度抽象"著称，削弱受监管系统所需的可调试性与透明度 |
| **CrewAI** | 角色化多智能体、上手最快 | **不选**：控制力弱、token 开销最重（简单任务约他者 3×）、偏 opinionated |
| **AutoGen / AG2** | 多智能体对话研究友好 | **不选作主干**：经历 Microsoft v0.4 重写与社区 AG2 分裂，生态不稳 |
| **LlamaIndex** | RAG/数据索引检索强 | **边缘选用**：作 RAG 组件，非整体运行时 |
| **Semantic Kernel** | .NET 生态 | **不选**：与 Python 技术栈不匹配 |
| **Claude Agent SDK** | Anthropic 原生、生产就绪 | **借鉴不主用**（理由见 第 2.4 节） |
| **MCP（协议）** | 工具/上下文接入事实标准 | **采用**：集成层（ATS/HRIS/日历/邮件）以 MCP 工具暴露 |
| **可观测/评测**（Langfuse / OTel GenAI / promptfoo） | 追踪、评测门禁 | **采用**：受监管领域必须有步骤级追踪与 eval 门禁；优先可本地部署者 |

### 2.6 分层架构模型（Layered Architecture）

把"agent 系统"拆成清晰分层，**每层各自选型**——混淆这几层正是"要不要用框架"困惑的根源：

| 层 | 解决什么 | 选型 | 理由 |
|---|---|---|---|
| **A. Agent 运行时 + 记忆** | 单任务内"思考-调工具-记忆"（如"筛这 50 份简历"） | **Hermes 派生内核**（拥有） | 记忆是护城河、要可审计、要拥有、本地优先 |
| **B. 长流程编排** | 跨天、跨多人、可暂停/续跑的业务流程（整个招聘 loop） | **自研轻量状态机 → 复杂后升级 Temporal/LangGraph** | Hermes 的会话循环非业务流程引擎；这是缺口所在 |
| **C. 集成** | 接 ATS/日历/邮件 | **MCP + 少量手写连接器** | 只需几个、要强控制，不需要 LangChain 几百个连接器 |
| **D. RAG / 检索** | 简历/JD/知识库检索 | **LlamaIndex 或直接向量库 + 嵌入式本地向量库** | 专用库即可；本地优先选嵌入式 |
| **E. 可观测 / 评测** | 追踪、eval 门禁 | **Langfuse / OTel / promptfoo** | 受监管必备；优先可本地部署 |

> **关键区分（A 层 vs B 层）**：Hermes 内核解决"单次 agent 推理 + 记忆"（A 层）；而招聘流程是**跨天、多方、可续跑的业务流程**（B 层），需要状态机/编排能力。二者是**互补的层**，不是竞争关系——B 层按需引入编排件，不动 A 层的"拥有主干"原则。

### 2.7 内核构建策略：移植 vs 重写（Build Strategy）

**许可证结论**：Hermes 采用 **MIT 许可**（Copyright 2025 Nous Research）。MIT 允许在闭源商用产品中自由使用、修改、再分发、销售，唯一义务是在**复制的实质性部分**保留 MIT 版权与许可声明。**因此"移植代码"在法律上是干净的**（商用无碍）。

**决策尺子**：

> **该移植（lift 代码）**：高价值 + 边界清晰 + 自研很难 + 安全关键。
> **该重写（借鉴设计）**：通用件 + 与 Hermes 自家产品强耦合 + 需要干净的服务形态与所有权。

| Hermes 部件 | 策略 | 理由 |
|---|---|---|
| **记忆子系统**（`memory_tool.py` / `memory_provider.py` / `memory_manager.py`） | **移植代码**（lift + 改造） | 护城河、边界干净、自研最贵——MIT 允许直接拿 |
| **注入防御/围栏**（`threat_patterns.py`、`StreamingContextScrubber`） | **移植代码** | 安全关键、易写错、已打磨 |
| **Agent 循环 / 运行时**（`conversation_loop.py` 等） | **借鉴设计、重写精简版** | 循环概念简单；Hermes 版与其 CLI/gateway 强耦合；需本地优先、干净所有权 |
| **gateway / CLI / TUI / 多 provider 管线** | **不拿，按需自建** | 属其"产品"部分，非本产品所需后端 |

**工程/法律卫生（移植代码时）**：① 保留 MIT 版权与许可声明（置于 `NOTICE`/`THIRD_PARTY` 文件）；② 移植代码须**自做安全审查**（MIT 为 "as is" 无担保，受监管产品不可盲信）；③ 连带核查被移植模块的**传递依赖**各自许可证。

> **ADR 摘要**：自建主干 = **Hermes 派生 Agent 内核 + 记忆子系统**；模型 = **本地优先、provider 无关、可选 Claude**；集成 = **MCP**；编排（B 层）确有需要时引入 **Temporal/LangGraph**；RAG 用 **LlamaIndex/嵌入式向量库**；可观测用 **Langfuse/OTel**。**不**用 CrewAI/AutoGen 作主干，**不**把产品建在 Claude Code/Agent SDK 运行时上。
> **待验证（Phase 0 spike）**：Hermes 内核移植工作量与边界；本地模型可用性/质量/硬件要求；嵌入式向量库规模上限；M3 是否需要 Temporal/LangGraph。

## 3. 背景与问题陈述（Problem Statement）

| 模块 | 现状痛点 | Jobpin Agent 的切入 |
|---|---|---|
| 简历匹配 | 关键词匹配召回差、漏掉转述能力、无法解释"为什么匹配"、招聘官淹没在海量简历中 | 语义+结构化混合匹配 + 可解释评分 + 组织校准 |
| 人才搜索 | 跨渠道（LinkedIn/内部库/历史候选人）割裂、被动等投递、Boolean 检索门槛高 | 自然语言→多源搜寻 agent，主动召回沉睡候选人，记住"找过谁、为何拒" |
| 招聘 | 流程在 ATS/邮件/日历/IM 间断裂，安排面试耗时，反馈难收敛，候选人体验差 | 端到端流程编排 agent + 面试反馈结构化 + 校准会议辅助 |
| 培训 | 培训与岗位能力脱节、千人一面、学完即忘、无法追踪能力增长 | 基于能力图谱的个性化学习路径 + 员工记忆驱动的持续赋能 |
| 监督与 KPI/考勤 | 目标-绩效-考勤数据孤岛、主观偏差、合规与隐私高风险、"监控"伤害信任 | 目标对齐与 KPI 洞察（**辅助而非自动处罚**）+ 强人工监督 + 透明可申诉 |

**根本痛点**：HR 系统不缺数据，缺的是**跨时间的记忆与上下文**和**能执行多步骤工作的智能体**。这正是 Hermes 记忆架构的价值所在。

**一个被忽视但最尖锐的细分——"没有 HR 的公司"**：大量中小企业/初创**雇不起专职 HR**，创始人/运营者既不熟专业招聘流程，也不懂澳洲雇佣合规（APP、反歧视、Fair Work、工作场所监察），却仍要招人、带人、做绩效。对他们，市面工具要么假设"已有 HR 专家"，要么只是任务自动化。**Jobpin Agent 把专业流程与合规护栏内建为产品本身**，让非 HR 的人也能把 HR 做对——这是本产品最强的价值主张与营销切入（设计要求见 第 11.8 节）。

**为什么不是现有方案（替代分析）**
- **通用 LLM / ChatGPT**：无组织记忆、无流程编排、无合规护栏，且把 PII 发往云端——记不住本公司标准，也不解决合规与隐私。
- **传统 ATS/HRIS（Greenhouse/Workday 等）**：是记录系统（system of record），非智能体；流程仍靠人；SMB 多半没有、也养不起。
- **点状 AI-HR 工具**：多为单点功能（如筛简历），缺统一记忆与全生命周期编排，少做本地优先 + 澳洲合规。
- **Jobpin Agent 的位置**：本地优先 + 组织记忆护城河 + 内嵌专业性与合规的全生命周期智能体——尤其填补"无 HR"中小企业的空白。

## 4. 目标与非目标（Goals & Non-Goals）

### 4.1 目标（可度量）
> **说明**：下列量化目标为**试点待验证的假设（非承诺）**，须先测定人工基线再对比；最终阈值在 Phase 1 依基线确定。
- **G1**：MVP（招聘前段）使单个职位的"JD→优质候选人 shortlist"周期缩短 ≥ 50%。
- **G2**：简历匹配 Top-10 命中率（招聘官认可占比）≥ 人工基线，并提供 100% 可解释评分。
- **G3**：每一项影响个人的决策建议都有 **HITL 复核入口 + 可解释理由 + 审计记录**（合规硬指标；为 Privacy Act ADM 透明义务做准备）。
- **G4**：偏见审计（adverse-impact ratio 等技术指标 + 间接歧视风险审查）在每个上线模块达标，产出可披露的审计报告（最佳实践，对齐澳洲反歧视法与自愿性 AI 指南）。
- **G5**：记忆系统在 10 万级候选人 / 万级员工规模下，相关上下文召回 P95 < 800ms（本地硬件上），且支持数据当事人级清除（APP 11.2 销毁/去标识、APP 13 更正）。
- **G6（"无 HR"细分）**：让无 HR 背景的用户在系统引导下完成合规的招聘/入职/绩效流程——关键操作有合规护栏 + 通俗解释 + 安全默认值，高风险动作自动提示并可一键升级到专业人士。目标：核心 HR 流程合规校验通过率 ≥ 设定阈值、关键步骤遗漏率显著下降。
- **G7（本地优先）**：商用 MVP 的 agent 运行时、记忆与 HR 数据默认在客户本地运行/存储；候选人/员工 PII 默认不离开客户机器（支持"完全本地"模式）。

### 4.2 非目标（YAGNI，明确不做）
- **N1**：MVP 不做全自动拒绝/录用/解雇决策（永远 HITL；监督模块不做自动处罚）。
- **N2**：MVP 不做云端多租户/计费/自助注册（保留干净的 org 抽象，作 Phase 4 可选后续）。
- **N3**：不自建 ATS/HRIS 核心系统，而是**集成**（Workday / SuccessFactors / Greenhouse / BambooHR 等）。
- **N4**：不做面部识别/情绪识别/键盘记录类侵入式监控（合规与伦理红线，直接排除）。
- **N5**：不替代法务判断——所有合规设计需澳大利亚法律顾问最终确认；内嵌的是护栏与引导，非法律意见替代。
- **N6**：试点仅限澳大利亚，不在本期处理澳洲以外法域合规。

## 6. 成功指标（Success Metrics / KPIs）

**North Star（按细分）**：① 有 HR/高量场景——每位用户每周产出的"被用人经理接受进入面试的优质候选人"数量；② **"无 HR" SMB（主力细分）——端到端完成一次合规招聘/入职且无需升级到外部专家的比例与用时**（高量吞吐指标对一年招几次的 Dana 不适用）。

## 9. Memory Subsystem ★（本产品核心，移植并扩展 Hermes 记忆系统）

> 本章先**讲清 Hermes 记忆系统怎么设计的、为什么越用越好**（第 9.1 节、第 9.4 节），再给移植映射（第 9.2 节）、本产品分层（第 9.3 节）、HR 治理（第 9.5 节）与验收（第 9.6 节）。

### 9.1 Hermes 记忆机制详解（加载 / 存储 / 优化 / 压缩 / 移除）

**两态模型**：内置 `MemoryStore` 维护两份并行状态——
- **冻结快照 `_system_prompt_snapshot`**：在 `load_from_disk()` 时一次性生成，注入系统提示，**整个会话不再变**（保住 prefix/prompt 缓存稳定，省 token/成本）。
- **活动态 `memory_entries` / `user_entries`**：被工具调用实时修改并落盘；工具响应始终反映活动态。

**两个文件、以分隔符（`§`，即 Hermes 源码中的 `ENTRY_DELIMITER`，独占一行的章节号）分隔、定长预算**：`MEMORY.md`（agent 笔记）与 `USER.md`（用户画像），条目以 `§`（`ENTRY_DELIMITER`）分隔；各有字符预算（默认 2200 / 1375），强制"少而精"。

| 生命周期 | Hermes 实际机制（代码层） |
|---|---|
| **加载 Load** | `load_from_disk()`：读两文件 → 按 `§` 切分/去空 → **去重**（`dict.fromkeys`，保序保首条）→ 逐条用 `threat_patterns`（strict）**威胁扫描**，命中则在**快照**里替换为 `[BLOCKED: …]` 占位（活动态保留原文供用户查看/删除）→ **冻结快照**。扫描对磁盘字节确定性 → 快照整会话稳定。 |
| **存储 Store** | `save_to_disk()` → `_write_file()`：写临时文件 → `fsync` → **原子 `os.replace`**（读者只会看到完整旧文件或完整新文件，无截断竞态）；读改写在 **独立 `.lock` 文件**的排他锁下进行；写前对 add/replace 内容做注入/外泄扫描。 |
| **优化 Optimize** | ① 去重；② **冻结快照**最大化 prompt 缓存命中；③ **定长预算**逼迫高信噪；④ 成功响应**刻意精简**（不回显全部条目，防模型"再找点改"的抖动）；⑤ `apply_batch`：多操作**全有或全无**、只对**最终预算**校验——一次工具调用内"先腾挪再新增"，免去多轮重发上下文。 |
| **压缩 Compress** | Hermes 提供压缩前钩子 `on_pre_compress(messages)`，外部 Provider 可返回"应保留的关键事实"。**重要更正：在 Hermes 当前主干中该返回值仅作通知、未自动并入压缩摘要（`conversation_compression.py` 调用处丢弃返回值），且内置文件型 `MemoryStore` 不是 Provider、无此钩子。** 故"压缩不丢关键信息"在 Hermes 中是**可用扩展点而非现成保证——Jobpin Agent 需自行接线**（把钩子产出注入摘要/记忆）。定长预算本身则是一种"持续压缩/策展"。 |
| **移除 Remove** | `remove`/`apply_batch`：按 `old_text` **子串匹配**删除；若匹配到**多个不同条目**则报错要求更具体（防误删）；**外部漂移检测** `_detect_external_drift`：发现"无法 round-trip 的内容"或"超限的巨型条目"（疑似外部写入）→ 先快照 `.bak` 再**拒绝本次写入**（防静默数据丢失）。 |

### 9.2 移植映射表（Hermes → Jobpin Agent）

| Hermes 机制 | 直接移植 | HR 化改造 |
|---|---|---|
| 内置文件型 `MemoryStore`（定长/原子/锁/漂移/备份） | ✅ 移植代码 | 承载**组织记忆**与**招聘官偏好**（低频、需精编、强一致的小体量层） |
| 抽象接口 `MemoryProvider`（`initialize/prefetch/sync_turn/on_pre_compress/on_session_switch/...`） | ✅ 接口契约移植 | 实现 `CandidateMemoryProvider`/`EmployeeMemoryProvider`/`OrgMemoryProvider`/`SemanticRAGProvider` |
| 回合前 `prefetch` / 回合后 `sync_turn`（后台单 worker 串行，保证 turn N 先于 N+1） | ✅ 移植 | 回合前召回相关历史；回合后异步落库；串行保证一致性 |
| 系统提示**冻结快照** | ✅ 移植 | 大规模省成本；组织政策/合规约束作稳定前缀 |
| 写入**威胁扫描** + `<memory-context>` 围栏 + 流式清洗 | ✅ 移植并**强化** | 简历/邮件 prompt-injection 是真实攻击；外部文本进记忆/上下文前强制扫描+围栏 |
| 去重/批量合并（`apply_batch`/`dict.fromkeys`） | ✅ 移植 | 记忆卫生：合并重复、原子化更新候选人画像 |
| "同一时刻仅一个外部 Provider" | ⚠️ **放宽** | HR 需多专用 Provider 并存 → `CompositeMemoryProvider`（按实体/查询路由并归并） |

### 9.3 Jobpin Agent 记忆分层（Memory Layers）

```
工作记忆 Working      —— 单次会话草稿（会话级，易失）
       │
实体记忆 Entity       —— 长期、结构化+语义，按实体聚合：
   ├─ 候选人 Candidate （技能/经历/历次互动/面试反馈/偏好/同意/溯源）
   ├─ 员工 Employee    （角色/能力/OKR/培训史/绩效信号/1:1 要点/成长）
   ├─ 职位 Job/Req     （需求/理想画像/校准结论/历史成效）
   ├─ 组织 Org         （招聘标准、能力框架、评分标尺、历史决策、政策）★护城河
   └─ 偏好 Recruiter/Mgr（个人偏好/沟通风格/用人经理的"bar"）
       │
语义/情节 Semantic    —— 互动/简历/JD/反馈向量化（嵌入式本地向量库），支撑 RAG
       │
程序性 Procedural     —— 学到的工作流/playbook（"怎么跑一个 SWE 招聘 loop"）+ 第 11.8 节 合规流程知识
```

- **小体量、需精编层**（Org、Recruiter）→ 移植的文件型 `MemoryStore`（定长、原子、强一致、可人工审阅 diff）。
- **大体量、需检索层**（Candidate、Employee、Semantic）→ **嵌入式本地向量库** + 本地结构化库，封装在统一 `MemoryProvider` 接口后，复用 Hermes 的 prefetch/sync/生命周期编排。

### 9.4 为什么"越用越好"——记忆学习闭环

```
      ┌─────────────────────────────────────────────┐
      │  每个回合 / 每次决策                          │
      │                                              │
   回合前 prefetch ──► 召回相关实体记忆（围栏注入）   │
      │                      ▼                       │
      │             agent 推理 + 行动 + 给出建议      │
      │                      ▼                       │
   用户反馈（采纳/否决/更正）+ 新事实                  │
      │                      ▼                       │
   回合后 sync_turn ──► 异步落库（候选人/组织记忆）    │
      │                      ▼                       │
   组织记忆校准（"本企业眼中的好 X 标准"持续更新）   │
      └──────────────► 下次召回更准 ─────────────────┘
   （长会话压缩前 on_pre_compress 抽取关键事实，确保不丢）
```

机制上之所以**越用越好**：① 每次完成的回合把**新事实**写回记忆（`sync_turn`），下次 `prefetch` 再召回——知识单调累积；② 招聘官的**采纳/否决反馈**写入组织记忆，使匹配校准逼近"本企业真实用人标准"（learning-to-rank）；③ **定长+去重+漂移防护**保持高信噪；④ Provider 接口让记忆后端可独立升级。

**适用范围与两个诚实前提**：(a) learning-to-rank 飞轮**需要反馈量**——对一年招几次的"无 HR" SMB，per-org 校准短期不会显著收敛，**该细分的 day-1 价值主要来自下方"冷启动"的内建专业知识，而非校准飞轮**；"越用越好"的强曲线主要适用于高量招聘者。(b) 把偏好/招聘官"bar"写入组织记忆可能**放大个人偏好甚至偏见**，与 第 9.5 节"禁用受保护属性/代理变量"存在张力——故须设**反馈偏见放大控制**：对写入校准的偏好做受保护属性/代理变量扫描，并纳入偏见审计监控。**这把"无 HR 痛点"也接上了**：专业 HR 流程与合规规则作为**程序性/组织记忆**沉淀（第 11.8 节），系统对该公司越来越"像一个懂行的 HR"。

**冷启动（day-1 价值）**：组织记忆为空时，系统以**内建的专业 HR 流程与澳洲合规知识**（第 11.8 节）作为程序性/组织记忆的"出厂种子"——day-1 即有专业基线可用，再随使用把该公司的具体标准沉淀进来（**先有用、后变强**），从而避免"记忆护城河"在初期落空。

### 9.5 HR 特有的记忆治理（Memory Governance）★合规关键（澳大利亚）

| 治理能力 | 设计 | 合规驱动（澳洲） |
|---|---|---|
| 租户/实体命名空间 | 记忆 key = `tenant : org : entity_type : entity_id` | 隔离、最小权限、APP 11 |
| 来源溯源 Provenance | 每条记忆记来源，可回链原始证据 | 可解释/可申诉、可审计、ADM 透明（APP 1） |
| 合法性/同意标签 | 收集目的/同意/用途标签 | APP 3/5/6 |
| 保留期 / TTL | 候选人数据按政策过期；录用/未录用分策略 | APP 11.2 |
| 数据当事人级清除/更正 | 从**活动库**删除该个人全部记忆+派生向量并**去标识**；备份不做即时级联，而按**保留期到期自然老化**（清除承诺限定于活动存储） | APP 11.2（销毁/去标识，非 GDPR 式即时擦除）、APP 13、投诉处置 |
| 记忆 RBAC | 仅能召回授权范围内的记忆 | 最小权限、APP 11 |
| 全量审计（读&写） | 记录 who/what/when/why（本地审计日志） | 可问责、NDB 取证 |
| 偏见卫生 | 禁存/禁用受保护属性作决策特征，扫描代理变量 | 联邦反歧视法、AHRC |
| 记忆可解释 | 任一建议可展开"基于哪些记忆事实" | 信任、可申诉、ADM 透明 |

### 9.6 记忆系统验收标准
- 召回相关实体上下文 P95 < 800ms：**默认基准为 SMB 真实规模（数百–数千候选人）于已发布最低硬件分档**；10 万级为单列压力测试上限，目标按硬件分档给出。
- 数据当事人级清除：活动库（结构化+向量+缓存）内即时删除/去标识；备份按保留期老化（见 第 9.5 节），不承诺即时级联。
- 注入测试：1000 条"对抗性简历/邮件"全部被围栏，0 例指令被执行。
- 记忆写入 100% 带溯源与合法性标签；无标签写入被拒绝（继承 Hermes"无效写入即拒绝"）。

## 10. 模块功能需求（Functional Requirements by Module）

> 每模块给出：场景、功能需求、Agent 工具、AI 方法、关键边界/合规、度量。

### M1 — 简历匹配（Resume Matching）
**场景（招聘官 / Dana）**：给定一个 JD，系统从候选库/投递中找出最匹配的人，并解释每人为何匹配/不匹配，以便快速决定。
- F1.1 简历解析：PDF/Word/纯文本/LinkedIn 导出 → 规范化结构（技能/经历/教育/证书）。
- F1.2 JD 解析与"理想画像"：从 JD + 组织记忆校准 must-have / nice-to-have / 反信号。
- F1.3 混合匹配：语义（embedding）+ 结构化（技能/年限/地点/工作权利）+ 组织校准。
- F1.4 **可解释评分**：分项打分 + 自然语言理由 + 证据引用（溯源到简历原文）。
- F1.5 去偏：屏蔽受保护属性与代理变量；姓名/性别/年龄/照片可选匿名化（blind screening）。
- F1.6 反馈回路：采纳/否决 → 写入候选人/组织记忆，持续校准（learning-to-rank）。

**Agent 工具**：`parse_resume`、`parse_jd`、`match_candidates`、`explain_match`、`anonymize_profile`。
**AI 方法**：本地 embedding 检索 + 重排 + LLM 结构化抽取与解释（模型分层：轻量本地模型抽取、强模型解释/打分）。
**边界/合规**：F1.4 解释 + HITL = 为 ADM 透明（APP 1）与可问责做准备；F1.5 去偏 + 偏见审计 = 降低间接歧视风险（反歧视法、AHRC）；解释必须**接地**（禁止幻觉资历）。
**度量**：Top-10 命中率、解释被认可率、各群体通过率比。
**"无 HR" SMB 的 day-1（冷库）行为**：此类客户**没有候选人库**——M1 在仅有少量邮件/文件简历时即可工作：解析 → 用**组织记忆的内建专业基线**（第 9.4 节 冷启动）+ JD 校准出理想画像 → 对手头这几份做可解释排序与缺口说明。价值来自"专业筛选 + 解释 + 合规护栏"，而非大库检索。

### M2 — 人才搜索 / 主动寻源（Talent Search / Sourcing）
**场景**：用一句话描述要找的人，系统从内部库、历史候选人、（授权的）外部渠道主动召回，并记住找过谁、为何放弃。
- F2.1 NL → 多源搜寻：内部库、历史候选人、（授权 API）LinkedIn/招聘平台。
- F2.2 "沉睡候选人"召回（**主要面向"有 HR"/已有历史候选库的客户；"无 HR" SMB 冷库下不适用，列为后续**）：基于候选人记忆主动唤醒过往合适者（合规重联）。
- F2.3 布尔/向量混合检索 + 相似扩展。
- F2.4 搜寻 agent 编排：拆解→并行检索→去重归并→排序→记忆"搜寻轨迹"。
- F2.5 外联草稿：个性化触达草稿，**人工确认后发送**（不自动群发）。

**边界/合规**：外部渠道须**授权 API**（禁违反 ToS 的爬取）；联系候选人符合 APP 5 与电子营销同意（Spam Act）；外联非自动发送。

### M3 — 招聘流程编排（Recruitment Workflow）
**场景（招聘官/用人经理/Dana）**：从筛选到 offer 的流程由 agent 编排，减少在 ATS/邮件/日历/IM 间来回。
- F3.1 流程状态机（**B 层**：自研轻量状态机起步，复杂后评估 Temporal/LangGraph），与 ATS 双向同步。
- F3.2 面试排程 agent：协调日历/时区/面试官负载，生成日程草案（人工确认）。
- F3.3 结构化面试反馈：competency-based 评分卡，自动汇总与冲突标注。
- F3.4 校准辅助：跨面试官对齐标准、识别评分偏差，写入组织记忆。
- F3.5 候选人沟通：状态通知、个性化（含尊重的拒信），人工把关。
- F3.6 候选人隐私门户：查看状态、行使 APP 12（访问）/ APP 13（更正）及投诉。

**边界/合规**：每个"影响候选人"节点都有 HITL；评分卡基于岗位相关能力；F3.6 满足 APP 12/13 与投诉处理。

### M4 — 员工培训（Employee Training / L&D）
**场景**：按岗位能力要求与当前缺口，生成个性化学习路径并追踪成长。
- F4.1 能力图谱：岗位 → 能力 → 学习资源。
- F4.2 缺口分析：员工记忆（现有能力）× 岗位要求 → gap。
- F4.3 个性化学习路径（内部课程/外部/导师/项目历练）。
- F4.4 学习陪伴 agent：答疑、知识检索（RAG over 内部知识库）、阶段测评。
- F4.5 成长追踪：前后测、能力增长、与绩效相关性，写入员工记忆。
- F4.6 与招聘联动：内部流动/晋升时数据成为内部候选人画像（打通 M1/M2）。

**边界/合规**：员工数据用于发展目的需透明（APP 5）；测评若影响晋升/绩效，进入高敏感决策需 HITL + 解释。

### M5 — 监督与 KPI/考勤（Supervision & KPI/Attendance）★最高合规敏感
> **定位（红线）**：**目标对齐与绩效洞察的辅助工具**，**非监工**。**不做**侵入式监控（无键盘记录/截屏/面部/情绪识别）。**不做**自动处罚/解雇。所有影响个人的判断 = 强 HITL + 可解释 + 可申诉。

**场景（主管）**：把团队目标（OKR/KPI）与执行进展、考勤打通，获得"谁需要支持/谁被低估"的洞察以更好辅导团队。
- F5.1 目标管理：OKR/KPI 设定、级联对齐、进展追踪（数据来自已集成系统）。
- F5.2 KPI 洞察：基于客观、岗位相关指标，给趋势/异常/需关注信号（**解释优先于打分**）。
- F5.3 考勤集成：对接现有考勤/排班（聚合，**不新建监控**）。
- F5.4 1:1/辅导辅助：基于员工记忆，为主管准备话题/认可/发展建议。
- F5.5 绩效复核辅助：汇总多源证据辅助**人工**评估；系统不下结论。
- F5.6 公平与申诉：员工可见评估数据与口径，可申诉、可纠错（APP 13）。

**边界/合规（最高门槛；取决于试点所在州）**：
- **州工作场所监察/监控法（按州差异大，须按确认的试点州适配）**：NSW《Workplace Surveillance Act 2005》对计算机/摄像/追踪监察要求 **≥14 天书面通知 + 明确政策、禁隐蔽监察**（厕所/更衣室绝对禁区）；ACT《Workplace Privacy Act 2011》另有其通知/同意要求（**与 NSW 的 14 天规则不等同，须单独核对**）；VIC《Surveillance Devices Act 1999》规制**监听/光学/追踪设备的隐蔽使用**（非"通知期"制度，勿与 14 天通知并列）；QLD 及其余州/领地多以**一般性监察设备法**规制——"无专门工作场所监察法"**不等于无约束**。本模块**不引入新监控**、仅聚合既有系统。
- **NSW WHS（Digital Work Systems）2025/26 修订**：将 AI 工作分配与自动化决策纳入 WHS；若在 NSW，需评估 WHS（含心理健康）义务。
- **Fair Work Act 2009**：一般保护/不利行为/不公平解雇——**禁止任何自动化不利处置**；绩效动作须人工决策。
- **Privacy Act / APPs** + ADM 透明（APP 1）。
- **员工咨询**：按适用现代裁定/企业协议（award/EA）协商条款进行（澳洲无统一强制工会协商，但协商义务可能源于 award/EA 与良好实践）。
- **上线前置**：PIA + 法务签字 + 员工咨询 + 偏见审计，缺一不可（见落地计划 Phase 3 Gate）。
- **州决策矩阵（M5 是否/如何上线取决于试点州，第 14 节 待决#2）**：须为试点州明确——① 适用哪些监察/隐私/WHS 义务；② M5 默认**开/关**；③ 若该州无专门工作场所监察法，本模块仍按 Privacy/Fair Work/一般监察法从严、且默认**保守关闭**敏感能力。**"M5 在本试点被降级/暂不交付"是被允许、被规划的合法结果**，而非失败——Fair Work 与 Privacy Act 始终适用，与是否有州监察专法无关。

## 11. 跨模块需求（Cross-Cutting Requirements）

### 11.1 规范数据模型（Canonical Entities）
`Candidate, Job/Requisition, Application, Employee, Skill, Competency, Course/LearningResource, Goal/OKR, KPI, Review/Feedback, Interview, Interaction/Event, Consent, Org, User(role), AuditRecord, MemoryRecord`。
- 所有实体带 `org_id`（为未来多租户保留 `tenant_id` 抽象，MVP 单租户本地不启用多租户基础设施）。
- **审计**：追加写审计日志（本地）；MVP **不强制全量事件溯源**（见 第 13.1 节 简化原则）。

### 11.2 集成（Integrations）
- **MVP（"无 HR" SMB）最小集成**：本地**文件/邮件导入简历**（PDF/Word/邮箱附件/云盘）+ 日历与邮件——SMB 通常**没有** ATS/HRIS，故 MVP 不依赖企业 ATS。（**冷库现实**：此类客户也**没有历史候选库**，故 M1 以"少量简历 + 内建专业基线"工作、M2 沉睡候选人召回属"有 HR/后续"——见 第 9.4 节 与 M1/M2。）
- **HRIS/ATS（"有 HR"企业 / 后续）**：Workday、SAP SuccessFactors、Greenhouse、BambooHR、Lever（连接器框架，OAuth；以 **MCP** 工具暴露）。
- **寻源**：LinkedIn（官方/授权 API）、招聘平台。
- **协作**：Google/Microsoft 日历与邮件、Slack/Teams、SSO（OIDC/SAML、SCIM）。
- **设计**：统一连接器 SDK + 反腐层（anti-corruption layer）翻译为规范实体；Webhook + 轮询双向同步；速率限制与重试。集成为**出站、可选**调用（本地优先）。

### 11.3 AI / LLM 架构（本地优先）
- **模型策略（本地优先 + 分层）**：
  - **默认本地模型**（经 Ollama/llama.cpp 等运行开源模型，如 Llama/Qwen/Mistral 系列）处理 PII 重的抽取/分类/解析与多数 agent 推理 → 数据不出本地。
  - **可选云模型**（如 Claude Sonnet/Opus）用于高难推理/解释，**需显式开启 + PII 最小化/脱敏 + APP 8 处理**；可整体关闭进入"完全本地"模式。
  - **客户自带密钥 / 自有模型端点（BYO-key，推荐折中）**：本地应用以**客户自己的**模型账户（如客户自有的 Anthropic / Bedrock Sydney 密钥）直接调用云模型——数据从客户机器直达模型供应商、**不经本方任何服务器**：既得前沿模型质量，又使本方不接触 PII。此选项化解"本地硬件不足以跑高质量模型"的难题。**注意：BYO-key/可选云仍构成跨境披露——作为 APP 实体的客户对 APP 8 仍负责（本方不接触数据不免除客户义务）；缓解 = 选澳洲境内区域（如 Bedrock Sydney）+ 合同/APP 8.2 例外 + 出站脱敏。仅"完全本地"模式才真正不发生跨境披露。**
  - 模型路由器按任务难度/隐私级/硬件能力动态选择；provider 无关抽象避免锁定。**路由失败/密钥失效兜底**：长流程（招聘 loop）中云/BYO 调用失败或密钥被撤时，状态机暂存并回退到本地模型或挂起待人工，不丢流程。
  - **脱敏（de-identification）须有明确管线**（非口号）：出站前对 PII 检测 + 遮蔽/假名化，并在本地记录脱敏前后映射——这同时是 APP 8 的前置条件。
  - **可选模型供应商（可插拔适配层）**：provider 无关抽象以单一接口在其后为每个后端暴露一个适配器。**首发集合：OpenAI（首个落地的适配器，且为内部试点 / 开发的默认激活供应商，因已有账户）、Anthropic Claude、DeepSeek**，与本地模型路径并列。激活哪个供应商及其密钥属于**部署 / 客户配置（BYO-key）**而非改代码：Claude 与 DeepSeek 适配器按接口对齐构建，客户只需提供密钥即可切换。**这不改变商用产品的本地优先默认**（第 1、7 节、G7）：选用任一云供应商（OpenAI / Claude / DeepSeek）都会使 PII 出站，因而触发上文的显式开启 + 脱敏 + APP 8 管控；仅"完全本地"模式才不发生跨境披露。
- **Embedding/向量**：本地 embedding（如 BGE 系列）+ **嵌入式向量库**（sqlite-vec / LanceDB / Chroma 本地）。**嵌入模型须固定版本并随向量一同记录**；切换嵌入模型/维度会使向量空间不兼容，须走**重嵌入（re-embed）迁移**，不可静默混用。
- **Prompt 复用**：复用 Hermes"冻结快照"以稳定前缀、降本（对云模型尤甚）。
- **RAG**：混合检索（BM25+dense）+ 重排；引用接地、防幻觉（组件可用 LlamaIndex）。
- **Evals**：黄金集 + LLM-as-judge + 离线回归 + 在线 A/B；**偏见/公平 eval** 进 CI 门禁。
- **护栏**：输入侧 PII/注入检测；输出侧接地校验/敏感过滤/越权拦截。

### 11.4 安全与隐私（Security & Privacy，本地优先）
- 静态加密本地数据库与记忆文件（OS keystore/DPAPI/Keychain 管理密钥）；敏感字段加密。
- RBAC + ABAC（按 org/团队/角色/敏感度）；SSO + SCIM（企业场景）；最小权限。
- 追加写本地审计日志；密钥/secret 管理；DLP。
- **数据驻留**：默认数据/推理在本地（onshore 天然满足）；任何出站（云 ATS、可选云 LLM）走 APP 8 管控 + 最小必要 + 脱敏 + 可关闭。NDB 数据泄漏通报流程。
- 威胁建模 + 渗透测试 + 红队（含 prompt-injection via 简历）；本地应用的更新与完整性校验。
- **NDB（可通知数据泄漏）责任与 runbook**：本地模式下**客户是持有数据的 APP 实体、承担 30 天评估/通报义务**；无 IT 的 SMB 本地机器丢失/勒索风险高且本方无可见性。须提供：本地泄漏检测信号、客户侧"评估→通报 OAIC/个人"的引导工具、责任划分说明；**静态加密**作为 NDB 的补救/安全港。列为 Phase 1 合规交付物。

### 11.5 合规与负责任 AI（Compliance & Responsible AI）★澳大利亚
> 澳洲目前**无强制性高风险 AI 专法**（National AI Plan 2025 暂不立法，依赖现行技术中立法律 + 行业监管 + 自愿性指南）。本产品按 HR 高敏感场景**从严**设计并对齐自愿性指南。

| 框架 / 法律 | 关键义务 | 在 Jobpin Agent 的落地 |
|---|---|---|
| **Privacy Act 1988 + 13 APPs** | 收集告知（APP 5）、目的限制（APP 3/6）、安全（APP 11）、不再需要即销毁/去标识（APP 11.2）、访问与更正（APP 12/13） | 记忆治理 第 9.5 节、隐私门户、保留/TTL、本地审计 |
| **ADM 透明（APP 1，2026-12-10 生效）** | 对"显著影响个人的自动化决策"在隐私政策披露 | HITL + 可解释 + 决策日志 + 披露模板 |
| **APP 8 跨境披露** | 向境外披露 PII，**作为 APP 实体的客户始终负责**（含 BYO-key） | **仅"完全本地"模式不发生跨境披露**；可选云/BYO-key 选澳洲境内区域 + 合同/APP 8.2 + 脱敏；不夸大为"规避" |
| **联邦反歧视法 + AHRC**（种族/性别/残障/年龄四法） | 禁直接/间接歧视 | 去偏、偏见审计、岗位相关性、可解释 |
| **Fair Work Act 2009** | 一般保护/不利行为/不公平解雇 | M5 禁自动化不利处置；绩效动作人工决策 |
| **州工作场所监察法**（NSW/ACT/VIC） | 通知（14 天）、政策、禁隐蔽监察 | M5 硬 Gate（视试点州）；不新建监控 |
| **NSW WHS Digital Work Systems（2025/26）** | AI 工作分配/ADM 纳入 WHS | 若在 NSW，评估 WHS（含心理健康）义务 |
| **自愿性 AI 指南** | 透明、问责、风险管理、人工监督、记录 | 对齐 2025《Guidance for AI Adoption》（6 项实践）；其前身《Voluntary AI Safety Standard》的 10 项 guardrails 仍可作控制集参考。**注意：2024 年提出的"强制性高风险 AI guardrails"已被搁置，勿与之混淆。** |
| **Spam Act 2003 / Do Not Call** | 电子营销同意 | M2 外联合规（同意、退订） |

- **雇员记录豁免（employee records exemption）谨慎处理**：Privacy Act 对私营雇主处理**现/前雇员**的"雇员记录"设有豁免，可能部分覆盖 M4/M5 员工数据；但**不覆盖候选人**（招聘前），范围有争议且在审查中。**更重要：2025-06-10 生效的"隐私严重侵犯法定侵权"已对依赖该豁免构成现实约束**（尤其 M5 监控类）——故对员工数据施加 APP 级保护**部分是当前法律必要，而非仅最佳实践/面向未来**。产品不依赖该豁免。
- **隐私严重侵犯的法定侵权（2024 改革引入）**：过度的员工监控/监察可能触发——M5 的比例、透明、最小必要据此从严。
- **HITL**：所有"影响个人"的输出默认建议态，需人工确认，记录决策者与理由。
- **可解释性**：每个建议可展开"基于哪些事实/记忆/指标"，溯源到原始证据。
- **公平**：上线门禁含偏见审计；持续监控群体指标漂移。

### 11.6 可观测性与质量（Observability）
- Agent 步骤级追踪（tool 调用、子代理、记忆读写）；本地成本/延迟/质量仪表盘（Langfuse/OTel，优先可本地部署）。
- 评测看板（质量、公平、幻觉）；告警；灰度发布与回滚（应用更新维度）。
- **本地优先下的遥测边界（关键）**：默认**本地留存**；只有**经去标识、客户 opt-in** 的聚合指标才回传本方。**含受保护属性的公平/偏见指标在本地优先下默认不出本地**——故 第 6 节 中"任一恶化即阻断发布"的护栏指标分两类：**本地自评 + 客户侧报告**（供客户自身放量决策）vs **本方可见的聚合信号**（供产品级发布门禁）；文档须标明每个护栏指标属哪类，避免"承诺了却拿不到数据"的门禁。

### 11.7 国际化与可访问性
- 试点为澳大利亚：**英文（澳式）优先**；多语言为后续可选项（非本期目标）。多时区按需；WCAG 2.1 AA。

### 11.8 面向非专业用户的"专业性与合规内嵌"（Expertise & Compliance Embedded）★
对"没有专职 HR"的客户，产品不仅自动化任务，更要**替用户具备专业性**。设计要求：
- **引导式工作流**：把专业 HR 流程（结构化招聘 loop、合规入职、绩效复核）做成分步引导，而非交给用户一个空白工具。
- **对"人"的合规护栏**（不只是对 AI）：用户将出错时主动拦截/提示——如面试禁问项（年龄/婚育/健康/族裔等可能构成歧视）、JD 歧视性措辞、不合规解雇动作。
- **合规模板库**：JD、面试评分卡、offer/拒信、隐私告知（APP 5）、政策——默认即合规、可本地化。
- **通俗语言解释**：把"为什么这样做 / 这条法律意味着什么"讲成非专业用户听得懂的话。
- **安全默认值 + 升级路径**：默认走最稳妥、最合规的路径；高风险/边缘情形**明确提示超出工具范围并建议咨询专业 HR/法律**（与 N5 一致）。
- **记忆承载专业性**：上述流程知识与合规规则作为**组织/程序性记忆**沉淀（见 第 9.3 节、第 9.4 节），使系统对该公司越来越"像一个懂行的 HR"——把"无 HR 痛点"与"记忆护城河"直接打通。

> **⚠️ 本节是最高责任面，须按一等模块工程化（不是一句卖点）**：对非专业用户主动判断"这道面试题能不能问""这个解雇动作合不合规"，是**面向无力鉴别者的合规指导**，叠加 LLM 幻觉风险（第 13.3 节）即最大法律暴露面。故要求：① **合规规则库（golden set）+ 持续 eval**；② 规则库**须经澳洲雇佣法律师审定并签字、定期复审**；③ 明确**准确率/精确率目标**，并把**"自信而错误的护栏建议"列为 P0 反指标**（错误地说"可以问"比不提示更糟）；④ 失败模式分析 + 兜底：**默认保守 + 一律可一键升级到专业人士**；⑤ 责任/免责姿态与 N5 一致（护栏与引导，非法律意见替代）。

## 13. 系统架构（概览，本地优先）

**部署形态**：单一**本地应用**（桌面或本地服务器），自包含 agent 运行时、记忆、本地数据库、嵌入式向量库、（可选）本地模型运行时；仅按需做出站调用（云 ATS、可选云 LLM、可选托管候选人门户）。

### 13.1 MVP 简化原则（避免过度设计）
为避免企业级架构在本地 MVP 上过度复杂，刻意**推迟**以下重型设计（保留抽象、不提前建设）：
- **多租户基础设施** → 本地天然单租户；保留 `org_id`/`tenant` 抽象，不建多租户隔离/计费（Phase 4 可选）。
- **全量事件溯源** → MVP 用关系库 + **追加写审计日志**即可满足可审计；事件溯源留给规模化/云路径。
- **`CompositeMemoryProvider` 多 Provider** → MVP 用"文件型 MemoryStore + 单一嵌入式向量库"即够；多 Provider 归并在 M4/规模化时再启用。
- **5 个子代理** → MVP（M1–M3）先上 Sourcing/Screening/Scheduling 三个；Training/KPI 随模块上线。
- **重型编排引擎（Temporal/LangGraph）** → 先自研轻量状态机，复杂度上来再引入（第 2.6 节 B 层）。**但"轻量"不等于弱保证**：自研状态机须满足**最低持久化契约**（崩溃后可恢复、跨天暂停/续跑、对发邮件/建日程等外部副作用**幂等**），并作为 Phase 0/1 退出标准；达不到则提前采用 Temporal——合规所需的"每步可审计"依赖该持久化。

### 13.2 可扩展性路径（Scalability）
- **单机纵向**：嵌入式向量库（LanceDB/sqlite-vec）可处理数百万向量；本地模型按硬件分档；多数中小企业单机足矣。
- **本地横向**：组织内可选"本地服务器 + 多客户端"部署（共享单一本地后端，避免文件型记忆的多副本冲突）。
- **多设备/多副本的组织记忆同步是本地优先的真实限制**：两台各自安装的机器各有独立文件 MemoryStore + 向量库，**无内置合并/冲突解决**；故"多设备/团队协作"要么走"共享本地后端"单实例，要么走**可选云/托管形态**——纯多副本本地不提供组织记忆同步（这也意味着 第 1 节 的记忆壁垒在多副本本地场景不自动跨设备累积）。
- **可选云路径（Phase 4）**：对需要多设备/团队协作/托管的客户，提供云/混合形态——此时才引入多租户隔离、云向量库、横向扩展的 agent runtime（保留的抽象使迁移可控）。
- **产品级扩展**：本地优先产品"复制每个安装"即扩展，运营更简单；中心化更新/可观测通过应用回传（可选、脱敏）解决。

### 13.3 主要风险（Risks）
| 风险 | 等级 | 缓解 |
|---|---|---|
| HR AI 法律/合规敏感（尤其 M5） | 高 | 合规优先架构、按模块 PIA、法务 Gate、M5 最后做并设硬门槛 |
| 偏见/歧视风险 | 高 | 去偏、偏见审计进 CI、持续监控、HITL、岗位相关性 |
| 简历/邮件 prompt-injection | 中高 | 移植+强化 Hermes 威胁扫描与围栏；红队 |
| 本地模型质量不足影响匹配/解释 | 中高 | 模型分层；难任务走"客户自带密钥"云模型（数据不经本方）或脱敏云调用；持续 eval 选型；硬件分档 |
| 本地部署的运维/更新/备份门槛（尤其无 IT 的 SMB） | 中 | 一键安装/自动更新/自动本地备份；引导式运维；可选托管 |
| 记忆错误/陈旧污染决策 | 中 | 溯源、TTL、漂移检测、人工可纠错、记忆审计 |
| LLM 幻觉编造资历 | 中 | 接地引用、输出校验、eval 门禁 |
| 数据出境（用云 LLM API） | 中高 | 默认本地；用云时脱敏 + APP 8 + 可关闭 |
| 范围过大（5 模块） | 中 | 平台化共享底座 + 严格分阶段 + 第 13.1 节 简化 |
| 员工信任受损（M5 被视为监工） | 高 | 透明、辅助定位、可申诉、参与式设计、员工咨询 |
| **第 11.8 节 对非专业用户的合规护栏给出"自信而错误"的建议** | 高 | 律师审定规则库 + eval + 错误护栏作 P0 反指标 + 默认保守 + 一键升级专家（第 11.8 节） |
| 多用户并发写本地文件记忆（单 .lock/单 worker 瓶颈） | 中 | 明确 MVP 并发模型（单用户桌面 vs 共享本地后端）；多用户走单实例后端 |
| 本地优先下护栏指标拿不到数据（无法真正门禁） | 中 | 区分本地自评 vs 客户 opt-in 聚合回传；门禁只用拿得到的那类（第 11.6 节） |
| 无 HR SMB 验证被排到 Phase 4（核心假设最后才验） | 高 | 落地计划提前引入 SMB 设计伙伴 + 试点章程含 kill 标准 |

---

## 14. 假设与待决问题（Assumptions & Open Questions）
**假设**：商用 MVP **本地优先**；默认本地模型、可选云模型（provider 无关）；英文（澳式）优先；集成目标以企业现有 HR 栈为准（待确认）。

**待决（需相关方确认）**：
1. 企业现有 HR 技术栈（Workday？SuccessFactors？其他）→ 决定首批连接器。
2. **试点在澳大利亚哪个州/领地**（尤其 NSW？）→ 决定 M5 适用的工作场所监察法与 WHS 义务。
3. **本地硬件基线**：试点用户的机器能否跑所需本地模型？低配比例多大？→ 决定本地模型分档与"可选云模型"策略。
4. 是否允许任何 PII 出境到云 LLM（即便脱敏）→ 决定是否需要"完全本地"硬约束。
5. MVP 内部试点的目标岗位族。
6. 是否存在适用 award/EA 协商条款（影响 M5 员工咨询门槛）。
7. 团队规模与预算（决定落地计划时间线，见 `02-Production-Plan.md`）。
8. **MVP 并发模型**：单用户桌面，还是"共享本地后端 + 多客户端"？→ 决定鉴权、记忆 RBAC 与文件记忆并发处理。
9. **试点设计伙伴**：能否在 Phase 1 就接入一个"无 HR" SMB（核心商用假设）而非等到 Phase 4？→ 见落地计划试点章程。
